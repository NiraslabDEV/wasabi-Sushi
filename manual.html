01 / Entrega Fase 1

O site já está no ar.

Em uma única sessão de desenvolvimento acelerado, o Wasabi saiu de um protótipo HTML estático para um site profissional em produção, hospedado em servidor próprio, com banco de dados conectado e pronto para escalar.

Lembras do que foi pedido? Um site bonito do Wasabi com o cardápio completo, identidade do restaurante, contactos e delivery — para quem visita Vilanculos ou já é cliente local. Está feito. E foi montado em cima de uma base que aguenta crescer até virar PDV, sistema de reservas, delivery online e pedidos por QR code nas mesas — sem precisar refazer nada.

02 / Stack & Arquitetura

Como o site funciona por dentro.

Esta secção traduz cada peça do sistema em duas linguagens: a versão técnica (para quem for desenvolver depois) e a versão simples (para a Direcção do Wasabi entender o que está a pagar).

A pilha tecnológica em uma página

Frontend (o que o cliente vê)
Next.js 14 · React · TypeScript
Framework moderno usado por Nike, Notion, TikTok. Carrega rápido, é seguro, e tem suporte oficial da Vercel. TypeScript evita 80% dos bugs comuns.
Estilo & design
Tailwind CSS + design custom
Permite que o design fique 100% fiel ao protótipo original. O site mantém a identidade do Wasabi sem depender de templates genéricos.
Banco de dados
PostgreSQL + Prisma ORM
O mesmo banco usado por Spotify, Apple, Reddit. Prisma é a camada que torna seguro escrever/ler dados — preparado para reservas, pedidos, clientes.
Hospedagem
Railway (servidor + DB)
Servidor profissional na América do Norte. Sempre online, com backups, deploy automático quando o código é actualizado, e HTTPS incluído.
Código-fonte
GitHub (NiraslabDEV/wasabi-Sushi)
Todo o código vive em repositório versionado — histórico completo de mudanças, possível reverter qualquer alteração em segundos.
Build & deploy
Nixpacks · Node 22
Push no GitHub → Railway compila → site actualizado em ~2 minutos. Sem FTP, sem mexer em servidor manualmente.

Traduzido em linguagem do dia-a-dia

Versão técnica

O frontend é renderizado server-side via Next.js App Router, com componentes em React 18 tipados em TS strict. O CSS usa custom properties + Tailwind utility classes. Persistência via @prisma/client contra Postgres 16 do Railway, com output: standalone no build para imagem mínima.

O que significa na prática

O site abre rápido em qualquer celular ou computador. Toda a página é construída do servidor antes de chegar ao cliente — Google entende perfeitamente o conteúdo, o que ajuda no SEO. Os dados (reservas, futuramente pedidos) ficam guardados de forma segura num banco profissional.

03 / Estrutura do código

Onde fica cada coisa.

A organização do repositório foi pensada para ser óbvia. Cada secção do site é um ficheiro pequeno e independente — quem mexer no código (agora ou daqui a 3 anos) sabe exactamente onde ir.

Mapa do repositório

As perguntas mais comuns & onde mexer

$
"Quero mudar o preço do Combo Bazaruto"
Editar src/data/menu.ts — procurar "Bazaruto", mudar o número em price: 2500. Push no GitHub → 2 minutos depois está no ar.
$
"Quero adicionar um prato novo ao cardápio"
Mesmo ficheiro. Adicionar uma linha { name: "...", price: 000, desc: "..." } dentro da categoria certa. Tudo aparece automaticamente, incluindo nos filtros.
$
"Quero mudar o telefone do delivery"
Procurar "857277152" no projeto — está em Hero.tsx, Nav.tsx, Delivery.tsx, Visit.tsx e Footer.tsx.
$
"Quero trocar a foto do hero"
Em Hero.tsx — trocar URL da imagem. Pode usar imagens próprias colocando na pasta public/ e referenciando /foto.jpg.
$
"Algo quebrou em produção"
Rollback em 1 clique no Railway — volta para o deploy anterior em ~30 segundos. Nada se perde.

04 / Roadmap curto prazo

O que vem a seguir.

O sistema foi construído em fases evolutivas. A Fase 1 (esta) é só a fundação visível. As Fases 2 a 4 são as próximas — cada uma agrega valor real ao negócio sem refazer o que existe.

F1
Concluído

Site institucional + cardápio digital

Presença online profissional, cardápio sempre actualizado, contactos clicáveis.
  • 9 secções com design 100% fiel
  • 60+ itens organizados em 21 categorias
  • Hospedagem profissional + HTTPS + banco pronto
F2
Próximo · ~1-2 sem.

Reservas online com WhatsApp + e-mail automático

Formulário simples no site → mensagem chega no WhatsApp do restaurante + e-mail formal arquivado.
  • Cliente preenche: nome, telefone, data, hora, nº pessoas, observação
  • Sistema envia automaticamente: mensagem formatada para o WhatsApp do balcão (84 600 7007) + e-mail HTML para o gerente
  • Tudo fica registado no banco — futuro relatório de reservas/mês
  • Confirmação automática para o cliente por SMS ou WhatsApp (opcional)
  • Email transaccional via Resend (~$0/mês até 3.000 emails/mês)
F3
~3-4 sem.

Dashboard interno (gestão do dia-a-dia)

Painel privado, só para a Direcção e gerência — gerir tudo sem mexer em código.
  • Editar cardápio: adicionar/remover pratos, mudar preços, marcar "esgotado" em segundos
  • Reservas do dia/semana: lista visual, marcar como confirmadas, alterar estado
  • Estatísticas: nº de reservas/mês, dia mais movimentado, prato mais vendido
  • Login com permissões: Direcção (tudo), Gerente (cardápio + reservas), Caixa (só visualizar)
  • Sobe naturalmente em cima da base já criada — sem refazer nada
F4
~6-8 sem.

Delivery online com carrinho + pagamento

Cliente faz o pedido pelo site sem precisar ligar — entrega ou take-away.
  • Carrinho de compras com adicionar/remover itens, observações por prato
  • Cálculo automático de subtotal + taxa de entrega por bairro de Vilanculos
  • Pagamento: M-Pesa, transferência ou "pagar na entrega" (cash on delivery)
  • Notificação em tempo real para a cozinha quando entra pedido (som + popup)
  • Tracking do cliente: "preparando → pronto → a caminho → entregue"
  • Reduz drasticamente o tempo gasto a anotar pedidos por telefone

05 / Roadmap médio prazo

Quando o Wasabi virar plataforma.

A partir da Fase 5, o sistema deixa de ser "site com reservas" e passa a ser o cérebro operacional do restaurante. Mesas, garçons, cozinha — tudo conectado em tempo real.

F5
~3-4 meses

QR Code nas mesas — cliente pede pelo celular

Cada mesa tem um QR fixo. Cliente abre câmara → vê o cardápio do Wasabi → adiciona ao pedido → confirma — tudo sem instalar nada.
  • Setup físico: QR codes plastificados nas mesas (mesa 1, 2, 3...) — custo único de impressão
  • Cliente: scan → cardápio abre no navegador → escolhe → "Chamar garçom" ou "Confirmar pedido"
  • Pedido chega directo na cozinha com o número da mesa, sem ambiguidade
  • Cliente acompanha: "pedido recebido → na cozinha → pronto"
  • Fecha a conta: chama garçom ou paga directo no app por M-Pesa
  • Reduz erros de comanda. Liberta garçom para atender melhor. Mesas giram mais rápido.
F6
~4-5 meses

App de Garçom no celular (PDV móvel)

O sistema também funciona para o lado oposto: garçom com celular vira ponto de venda móvel — tira pedido directo na mesa, envia para cozinha.
  • App instalável (PWA) no celular Android do garçom — sem precisar publicar na Play Store
  • Login por garçom: cada pedido fica associado a quem atendeu (comissão / desempenho)
  • Mesa → cliente → pedido: garçom escolhe mesa, vê o cardápio, marca itens, envia
  • Trabalha offline: se o Wi-Fi cair, o pedido fica guardado e sincroniza quando voltar
  • Modificadores: "sem cebola", "ponto bem passado", "extra wasabi" — escolhas rápidas
  • Conta dividida na hora: 4 pessoas dividem o combo igual ou por consumo

Como os dois lados convivem

A
Clientescan QR na mesa
B
OU Garçomtira pedido no app
C
Bancoregista o pedido
D

Por que isto vale o investimento? Um restaurante com 15 mesas que rotaciona 3x por noite faz ~45 contas/noite. Cada erro de comanda custa em média 200 MT (prato refeito ou comp). Reduzir erros em 50% = ~3.000 MT/semana só nisso. E o cliente que pede pelo celular tende a pedir 15-20% a mais (estudos de mercado em redes que adoptaram QR menu).

06 / Roadmap longo prazo

A ambição máxima do sistema.

Até onde esta base pode chegar? Aqui está o cenário completo — não é fantasia, é o que cada uma destas peças permite hoje em dia, e o que já existe em redes maiores.

F7
~6 meses

KDS (Kitchen Display System) + impressora térmica

A cozinha passa a ter uma tela própria — pedidos aparecem em cards coloridos por prioridade, com tempo de espera, e podem ser marcados como "pronto" com um toque.
  • Tela na cozinha (tablet de 10" ~3.000 MT, fixo na parede) — sem mais comandas em papel
  • Cards por prato: nome, mesa, observações, tempo decorrido, alerta vermelho se passar do prazo
  • Estações separadas: sushi vs cozinha quente vs bar — cada um só vê o que lhe compete
  • Impressora térmica opcional para imprimir comanda de cozinha + recibo do cliente
  • Métricas: tempo médio de preparo por prato — onde existe gargalo?
F8
~9-12 meses

Fidelidade, BI & multi-loja

Quando o Wasabi crescer (segunda loja em Maputo? franquia?), o sistema acompanha sem refazer nada.
  • Cadastro de clientes: cada pedido associa-se a um número de telefone — histórico de compras, prato favorito
  • Programa de pontos: cada 1.000 MT gastos = 1 ponto. 10 pontos = combo grátis
  • Cupões e promoções: "Quarta-feira combo 04 com 20% off" — agenda automática
  • Painel BI: facturação por dia/semana/mês, ranking de pratos, horários de pico, ticket médio
  • Multi-loja: cada filial tem o seu cardápio/preços/equipa — Direcção vê tudo num só painel
  • Integração contabilidade: exportar facturação para sistemas fiscais moçambicanos
F9
"Quando fizer sentido"

Integrações externas & expansão

A partir daqui é o limite da imaginação — tudo é possível com a base que já temos.
  • Integração com Yango / Bolt Food / Glovo — pedidos externos chegam no mesmo dashboard
  • Reservas integradas com Booking / TripAdvisor — turistas reservam direto
  • Stock de ingredientes: quando salmão acaba, "California de Salmão" some automaticamente do cardápio digital
  • App próprio do Wasabi publicado na Play Store / App Store (após validar PWA)
  • WhatsApp Business API: reservas e pedidos via conversa formal automatizada
  • Marketing automatizado: cliente que não vem há 30 dias recebe cupão de regresso
A pergunta certa não é "podemos fazer isto?" — podemos. A base técnica suporta tudo o que está nesta página. A pergunta é: "quando vale a pena cada peça?" Isso depende do crescimento do Wasabi, do volume de pedidos, e do orçamento disponível em cada momento. O plano é evoluir conforme o negócio cresce — não construir o que ainda não rende.
